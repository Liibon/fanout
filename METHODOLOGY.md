# Measurement Methodology

This document describes exactly how fanout produces latency numbers,
so that results can be reproduced and compared across runs.

---

## Clock source

All latency measurements use Go's `time.Now()` which on Linux resolves to
`CLOCK_MONOTONIC` via `clock_gettime(2)`. On macOS (used for development
only) it resolves to `mach_absolute_time`. Do not compare numbers across
OS families.

---

## Load generation

The load generator (`loadgen/`) implements an **open-loop Poisson process**:

| Parameter | Default | Meaning |
|-----------|---------|---------|
| `--qps`   | 100     | Target queries per second |
| `--warmup`| 500     | Requests issued before the measurement window |
| `--measure`| 5000   | Requests in the measurement window |

**Inter-arrival times** are drawn from `Exp(1/QPS)` (exponential with mean
`1/QPS` seconds), so arrivals are a Poisson process, not a fixed timer.

**Coordinated omission handling:** the latency clock for request *i* starts at
the *scheduled* send time, not the actual send time. If the system is
saturated and request *i* is delayed by 50 ms waiting for a prior request to
complete, those 50 ms are included in its latency. This matches the MLPerf
LoadGen open-loop methodology.

**Warmup:** the first `--warmup` requests are issued at the target QPS but
their latencies are discarded. This allows FAISS index caches, JIT
compilation, and TCP connections to reach steady state before measurement.

**Measurement window:** only the `--measure` requests that follow warmup are
recorded. Percentiles are computed over this window only.

**Invalid-run detection:** if the achieved QPS is < 95% of the target QPS
(indicating the system could not keep up), the loadgen prints `INVALID RUN`
and exits with status 1. Do not report numbers from invalid runs.

---

## Percentile computation

Latencies are sorted and percentiles are taken as:

```
p(N) = sorted[ceil(N/100 * len) - 1]
```

This is "nearest rank" percentile, which matches NumPy's default and Excel's
`PERCENTILE.INC`.

---

## Dataset reproducibility

The dataset is generated by `dataset/main.go` with a **fixed RNG seed (42)**
using Go's `math/rand` with a `rand.NewSource(seed)`. The generation
algorithm is:

1. Draw each component of each 128-dim vector independently from N(0,1).
2. Write in little-endian binary: `[int32 n][int32 dim][n x dim x float32]`.

The SHA-256 digest of the generated file is written to `dataset.bin.sha256`.
Leaves read this digest and abort if the file does not match. Any run where
the dataset file has been modified (wrong digest) is invalid.

## FAISS index parameters

| Parameter | Value | Notes |
|-----------|-------|-------|
| Index type | `IndexFlatL2` | Exact brute-force, no approximation |
| Distance metric | L2 (Euclidean squared) | |
| Sharding | per-leaf `n/N_leaves` vectors | Leaf `i` holds vectors `[i*S, (i+1)*S)` |
| Top-K aggregation | global merge at root | Sorted by L2 distance ascending |

`IndexFlatL2` is exact and does not approximate. This differs from ANN
indexes (HNSW, IVF) that trade recall for speed. The decision to use exact
search is intentional: the benchmark stresses the *fan-out latency* problem,
not the quality of the search algorithm. Using an ANN index would change the
workload character and make results incomparable.

---

## Synthetic leaf model

When `SYNTHETIC=true`, the leaf replaces FAISS with a sleep drawn from a
specified latency distribution. Parameters:

| Env var | Default | Meaning |
|---------|---------|---------|
| `SYNTHETIC_MU` | 2.5 | Log-mean of baseline distribution (ln ms) |
| `SYNTHETIC_SIGMA` | 0.6 | Log-std of baseline distribution |
| `SYNTHETIC_HEAVY_PCT` | 0.01 | Fraction of requests routed to slow distribution |
| `SYNTHETIC_HEAVY_MU` | 5.0 | Log-mean of slow distribution (ln ms) |
| `SYNTHETIC_HEAVY_SIGMA` | 0.3 | Log-std of slow distribution |

The baseline distribution models normal FAISS search. The slow distribution
models GC pauses, network retransmissions, and other straggler events.

With defaults, baseline p50 is roughly e^2.5 ms (~12 ms) and slow path p50
is roughly e^5 ms (~148 ms). 1% of requests hit the slow path. With
fan-out=16, the probability that at least one leaf hits the slow path per
root request is `1 - 0.99^16 = ~15%`, creating a pronounced tail.

---

## Epsilon for cross-hardware comparison

Two runs on different hardware should produce p50 and p90 within **+/-30%**
of each other when using synthetic mode (latency is sleep-based and relatively
hardware-independent). In FAISS mode, p50 and p90 vary proportionally to the
ratio of single-core FLOP/s between machines; expect **+/-2x** without
normalization. Do not directly compare FAISS-mode numbers across different
CPU generations without normalizing by a known micro-benchmark (e.g.,
single-threaded DGEMV throughput).

---

## Ramp artifact exclusion

The load generator discards warmup latencies and does not exclude from the
tail of the measurement window. The last K% of requests in the measurement
window are *not* excluded because the open-loop model does not accumulate a
request backlog at the end the way a closed-loop model does.
## Measurement Precision

Latency is recorded from the *scheduled* send time, not the actual send time.
This corrects for coordinated omission: a slow response cannot mask the queue
latency of requests arriving behind it.

The loadgen drives arrivals as an open-loop Poisson process so service time does
not affect inter-arrival intervals.
